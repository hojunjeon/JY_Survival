## 그날 상황

Cycle 선택 메커니즘을 다듬다가 갑자기 머리가 멈췄다. AI가 "Phase에 너무 많은 태스크가 있을 때 일부만 진행하는 법"을 설명하면서 "Cycle 분할", "Plan 파일 편집", "일시정지 + 분할" 세 가지 방법을 제시했다.

읽다가 의문이 들었다. **Step 0~5 파이프라인은 도대체 어느 단위로 도는 거지?** 분명 처음에 들었을 때는 이해한 줄 알았는데, "Cycle 선택"을 설명하는 순간 헷갈렸다. 태스크 1개에 파이프라인 1번? Cycle 1개에 파이프라인 1번? 페이즈 1개에 파이프라인 1번?

그동안 묵시적으로 Cycle 단위라고 생각하긴 했는데 명시적으로 확인한 적이 없었다. 모호한 상태로 다음 단계 가면 안 되겠다 싶었다.

## 뭐가 나왔나

AI가 다이어그램으로 정리했다.

```
Phase 11
├─ Cycle A: event-ui-emotional (fix)
│  └─ 파이프라인 1회 통과
│     ├─ Step 0: spec 1개 작성       ← Cycle 단위
│     ├─ Step 1: plan 1개 작성       ← Cycle 단위 (안에 T1~Tn)
│     ├─ Step 2: T1→T2→T3 순차 실행  ← 태스크 반복 (커밋 단위)
│     ├─ Step 3: Cycle 전체 리뷰     ← Cycle 단위
│     ├─ Step 4: Cycle 전체 QA       ← Cycle 단위
│     └─ Step 5: blog draft 1편      ← Cycle 단위
```

답은 **Cycle 단위.** Task는 Step 2의 내부 반복(커밋 단위). 이게 한 번에 깔끔히 정리됐다.

이유도 명확했다. Per-Phase는 spec 1개로 다 커버하기엔 너무 광범위. Per-Task는 태스크 1개마다 spec/plan/리뷰/QA/블로그 6개 파일 만들 거냐는 오버헤드 폭발. **Cycle 단위는 블로그 1편 = Cycle 1개와 자연스럽게 1:1 대응.**

---

## 그래서 어떤 결정을 했나

Cycle 단위로 못박았다. 그리고 깨달은 것 — **이걸 spec 문서에 명시적으로 적어야 한다.** 묵시적 가정으로 두면 또 헷갈린다.

설계 문서 (`docs/superpowers/specs/2026-04-25-workflow-refactor-design.md`) 에 "3-Layer 모델: Phase / Cycle / Task" 섹션을 박았다. 단위를 표로.
