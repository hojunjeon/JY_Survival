# [프롬프트 일기 #2] "테스트 먼저 써줘"를 처음 해본 날

---

## 그날 상황

JY Survival에서 플레이어는 SSAFY 학생이다. HP 100으로 시작해서 사방에서 몰려오는 버그 몬스터에게 맞으면 깎이고, 무기는 이벤트(동기 도움 요청)로만 얻는다. 이 모든 게 `Player.js` 하나에서 시작된다.

기획이 끝났다. 이제 코드를 짜야 하는데 어디서부터 시작해야 할지 몰랐다. `index.html`부터? `Player.js`부터? 게임 루프인 `Game.js`부터?

막막할 때는 AI한테 시작점을 잡아달라고 하는 게 낫겠다 싶었다.

---

## 입력한 프롬프트

`/superpowers:test-driven-development` 를 실행하고 이렇게 요청했다.

> *"Player.js 구현해줘. 위치, HP, 이동, 충돌박스 포함. 테스트 먼저."*

---

## 뭐가 나왔나

AI가 `Player.test.js`를 먼저 작성했다. 구현 파일은 그 다음이었다.

```js
it('초기 HP는 100이다', () => {
  expect(player.hp).toBe(100);
});

it('데미지를 받으면 HP가 줄어든다', () => {
  player.takeDamage(20);
  expect(player.hp).toBe(80);
});

it('HP는 최대 HP를 초과할 수 없다', () => {
  player.heal(50);
  expect(player.hp).toBe(100);
});
```

12개 테스트를 다 작성한 다음에야 `Player.js` 구현이 나왔다. `Math.max(0, ...)`, `Math.min(this.maxHp, ...)` 같은 경계 처리가 자연스럽게 들어가 있었다. 테스트에서 경계 케이스를 먼저 명시해뒀으니 구현이 그걸 따라간 것이다.

그리고 renderer를 생성자에 주입하는 패턴이 나왔다. 처음엔 왜 이렇게 하는지 몰랐는데, AI가 설명했다.

*"테스트에서 PixelRenderer에 의존하지 않으려면 renderer를 외부에서 주입받아야 합니다. 테스트에서는 renderer 없이 생성하면 폴백 렌더러가 동작합니다."*

직접 짰으면 `player.render()` 안에 그냥 스프라이트 그리는 코드 때려넣었을 것 같다. 나중에 스프라이트 교체할 때 난감해졌을 거고.

## 그래서 어떤 결정을 했나

모든 구현을 이 흐름으로 하기로 했다. 테스트 먼저, 구현 나중. Cycle 1 34개, Cycle 2 64개, 최종 273개까지 누적됐다.

테스트가 있으면 나중에 뭔가 바꿀 때 덜 무섭다는 걸 이때 처음 실감했다.
