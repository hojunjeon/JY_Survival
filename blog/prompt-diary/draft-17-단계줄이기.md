## 그날 상황

Phase 진입 시 "Cycle 분할은 누가 하는가" 물음이 나왔다. AI가 추천한 옵션:

> a. 메인 에이전트가 phase_feedback 읽고 Cycle 후보 추출 → 사용자 승인 → phase-status.md 등록

수긍하고 넘어가려다가 한 번 더 봤다. 그러면 흐름이 이렇게 된다.

```
브레인스토밍 → 메인 (Phase 분할) → Plan 에이전트 (Task 분할)
```

3단계. 그런데 Plan 에이전트가 어차피 분할 작업을 하잖아? 그러면 Plan 에이전트가 Phase → Cycle 분할도 함께 처리하면 안 되나? 그러면 2단계가 된다.

```
브레인스토밍 → Plan 에이전트 (Cycle 분할 + Task 분할)
```

물어봐야겠다.

## 입력한 프롬프트

```
계획 에이전트가 phase를 cycle로 분할하고 분할된 cycle마다 task로 분할하는 게 낫지 않아?
a대로 하면 브레인 스토밍 -> 메인 에이전트 (페이즈 분할) -> 계획 에이전트 이런 흐름인데
계획 에이전트가 사이클 분할까지 하면 브레인스토밍 -> 계획 에이전트로 바로 넘어갈 수 있잖아.
두 방식의 장단을 다시 한줄씩 비교해줘
```

## 뭐가 나왔나

AI가 1줄 비교를 해줬다. 그리고 트레이드오프를 노출했다.

> B를 택하면 brainstorming의 위치 문제가 생깁니다.
>
> - B 흐름: brainstorming → Plan agent (cycles + tasks)
> - 이때 brainstorming은 **Phase 전체** 대상 → "이벤트 UI + 무기 레벨업 + 융합무기 + 스테이지" 모두 한 번에 다룸
> - 결과: spec이 비대하거나 얕아짐

A는 단계가 길어 거추장스럽고, B는 brainstorming 깊이가 얕아진다. 둘 다 단점이 있었다.

그러더니 AI가 절충안을 제안했다 — **C: Brainstorming 에이전트가 Cycle 분할까지 담당**.

```
[Phase 진입]
  └─ Brainstorming 에이전트 1회 호출
     ├─ phase_feedback 읽기
     ├─ 사용자와 대화하며 Cycle 분할 합의
     └─ Cycle별 spec 파일 N개 작성

[Cycle별 반복]
  └─ Plan 에이전트 → Task 분할
  └─ Step 2~5
```

메인 처리 단계 0개. brainstorming 깊이 보존. spec 파일 분리.

---

## 그래서 어떤 결정을 했나

C 채택. Brainstorming 단계가 Cycle 분할 + spec N개 작성까지 담당.

그리고 깨달았다 — **"왜 이 단계가 필요해?"라고 묻는 것만으로 절충안이 나왔다.** 처음 추천(A)은 자연스러운 답이었지만 "단계 줄일 수 있지 않냐"는 압박을 넣자 더 나은 안이 떠올랐다.
