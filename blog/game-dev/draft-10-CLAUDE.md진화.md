# [JY Survival #10] CLAUDE.md는 코드다 — AI 지시서를 5번 고친 이유

> 시리즈: 김지윤의 디버그 서바이벌 개발일지
> 키워드: CLAUDE.md, AI 협업, 워크플로우, 지시서 설계

---

## 게임보다 먼저 설계해야 했던 것

이 시리즈를 쭉 읽었다면, 매 편마다 "Superpowers → Gstack → GSD → 코딩 → 문서화" 흐름이 반복된다는 걸 눈치챘을 거다.

AI가 알아서 이 흐름을 따른 게 아니다. `CLAUDE.md`라는 파일에 명시해뒀기 때문이다.

CLAUDE.md는 Claude Code가 세션을 시작할 때 자동으로 읽는 일종의 지시서다. 프로젝트 규칙, 실행 순서, 금지 사항을 여기다 써놓으면 AI가 그대로 따른다 — 이론상으로는.

실제로는 쓰면 쓸수록 구멍이 보였다. 세션마다 AI가 맥락을 잃거나, 내가 원하지 않는 타이밍에 커밋을 하거나, 승인도 안 받고 코딩을 시작하는 일이 반복됐다.

그때마다 CLAUDE.md를 고쳤다. 총 다섯 번.

---

## ① 세션이 바뀌면 AI가 기억을 잃는다

**망했던 순간**: Phase 2 개발 초반, 새 세션을 열었더니 AI가 "현재 어떤 작업 중인가요?" 라고 물어봤다.

이미 지난 세션에서 한참 얘기했던 내용이었다. AI는 대화 내역을 세션 간에 가져가지 않는다. 당연히 알고 있었지만, 막상 처음 겪으니 황당했다.

Phase 1이 끝날 즈음, 게임 기획 내용을 담은 `docs/design.md`와 진행 상태를 기록하는 `docs/phase-status.md`를 만들어뒀다. 그런데 AI한테 "세션 시작 시 이걸 읽어라"고 명시해두지 않으면 AI는 읽지 않는다.

**그래서 추가한 것**: CLAUDE.md 상단에 "세션 시작 시 필독" 섹션을 만들고 두 파일을 명시했다.

```
## 세션 시작 시 필독 (IMPORTANT)
- docs/phase-status.md — 현재 Phase 및 다음 작업 확인
- docs/design.md — 게임 전체 설계 확정본
```

이후로 새 세션을 열어도 AI가 "Phase 2 Cycle 3 진행 중이군요" 하고 바로 이어받는다. 파일 두 개가 AI의 단기 기억을 장기 기억으로 바꿔줬다.

---

## ② AI가 커밋 타이밍을 제멋대로 잡는다

**망했던 순간**: 기능 하나를 구현하다 말았는데 AI가 중간에 "커밋해도 될까요?"라고 물어봤다. 코드가 절반만 짜여 있는 상태였다.

반대로 어떤 날은 큰 기능을 하나 완성했는데 세션이 끝날 때까지 커밋 얘기가 없었다.

AI 입장에서는 "커밋 언제 해야 하는지" 규칙이 없으니까 그냥 적당히 판단한 거다. 나는 당연히 알고 있다고 생각했는데, AI한테는 '당연한 것'이 존재하지 않는다.

**그래서 추가한 것**: Git 커밋 규칙 섹션을 명문화했다.

```
## Git 커밋 규칙
- 각 사이클 완료 시 반드시 git commit 한다.
- 사이클 중간의 임시 작업은 커밋하지 않는다.
```

짧다. 근데 이 두 줄이 없을 때와 있을 때의 차이가 컸다. 규칙이 없으면 AI는 판단을 추측으로 채운다.

---

## ③ 태스크가 커지니까 AI 혼자 감당을 못 한다

**망했던 순간**: Phase 3에 들어서면서 한 사이클에 해야 할 작업이 5~6개씩 쌓였다. AI한테 "이거 다 해줘"라고 맡기면 중간에 뭔가 빠트리거나, 순서가 꼬이거나, 앞에서 만든 코드랑 뒤에서 만든 코드가 충돌하는 일이 생겼다.

컨텍스트 창 한계 문제이기도 하고, 태스크 간 의존성을 AI 혼자 트래킹하는 게 어렵기도 했다.

**그래서 추가한 것**: Phase 3부터는 "Subagent-Driven" 방식을 명시했다.

```
## 실행 방식 (Phase 3 이후)
- Subagent-Driven 방식을 기본으로 사용한다.
  - 태스크 하나당 서브에이전트 1개 디스패치
  - 서브에이전트 완료 후 결과 리뷰 후 다음 태스크 진행
```

태스크를 잘게 쪼개서 각각 독립된 서브에이전트에게 맡기고, 완료 후 내가 결과를 리뷰하는 구조다. AI 혼자 다 하게 두는 대신, 내가 중간 검문소 역할을 맡는다.

작업 품질이 눈에 띄게 올라갔다.

---

## ④ 피드백을 주면 AI가 바로 코딩을 시작한다

**망했던 순간**: `phase_3.md`에 수정 요청 사항을 적었더니 AI가 읽자마자 코드를 짜기 시작했다.

문제는 내가 적은 내용이 명확하지 않은 부분이 있었다는 거다. "배경 타일링 추가"라고 썼는데, AI는 이게 카메라 시스템과 연동되는 건지 독립적인 건지 판단을 잘못 내렸다. 결과물을 받고 나서야 "이게 아니었는데"가 나왔다.

중간에 확인 한 번만 했어도 방지할 수 있었다.

**그래서 추가한 것**: Phase 3+부터 `phase_N.md` 검토 단계를 필수로 삽입했다.

```
### [필수] phase_N.md 검토 단계
phase_N.md를 읽은 후 즉시 실행하지 않는다. 먼저:
1. 모호하거나 해석이 갈리는 항목 명시
2. 코드 분석으로 누락된 수정 항목 발굴 및 개선 방향 제안
3. 사람의 승인을 받은 후 실행 시작
```

이제 AI는 피드백을 받으면 먼저 "이 항목이 모호한데 이렇게 이해해도 될까요?"라고 묻는다. 코드 한 줄도 짜기 전에.

처음엔 한 단계가 더 생기는 게 번거롭게 느껴졌는데, 지금은 이 단계가 없으면 불안하다.

---

## ⑤ 파일 위치가 헷갈리기 시작했다

**망했던 순간**: Phase 3 작업을 마치고 나서, 루트에 있던 `phase_3.md`를 `phase_feedback/`으로 옮기는 절차가 있었다. 파일 이동이 안 됐거나 깜빡하면, 다음 세션에서 AI가 이미 완료된 Phase를 다시 진입하려는 일이 생겼다.

"루트에 파일 있으면 해당 Phase 진입" 규칙이 파일 이동에 의존하고 있었는데, 이동 단계가 실수 포인트였다.

**그래서 추가한 것**: 처음부터 `phase_feedback/` 에서 관리하도록 규칙 자체를 바꿨다.

```
### phase_N.md 관리 규칙
- 모든 phase_N.md 파일은 phase_feedback/ 디렉토리에서 관리한다.
- Phase 진입 조건: phase-status.md + phase_feedback/ 교차 확인
```

파일을 옮기는 게 아니라 처음부터 올바른 위치에 두는 방식으로. 실수할 여지를 없앴다.

---

## 결국 배운 것

CLAUDE.md를 고칠 때마다 공통된 패턴이 있었다.

AI가 이상하게 행동했다 → 규칙이 없어서 추측으로 채웠던 거다 → 규칙을 명시했다.

AI한테 일을 맡긴다는 건, 내가 '당연하다'고 생각하는 것들을 하나씩 명문화하는 과정이다. 처음부터 완벽한 지시서를 쓸 수 없다. 부딪혀봐야 어디가 구멍인지 보인다.

그리고 AI에게 더 많은 일을 맡길수록 지시서는 더 정교해져야 한다. 태스크가 커질수록, Phase가 쌓일수록, CLAUDE.md도 함께 자란다.

어떻게 보면 코드랑 똑같다. 처음엔 대충 짜고, 문제가 생기면 리팩토링한다.
