## 게임 시스템 설계하면서 결정이 어려웠던 것들

### 무기 시스템

무기를 기술 스택으로 표현한다는 건 처음부터 확정이었다. Python, C/C++, Java, Git, SQL...

문제는 **어떻게 새 무기를 획득하냐**였다.

뱀서처럼 레벨업 시 선택? 아니면 이벤트/퀘스트 클리어 보상으로만?

결국 후자를 선택했다. 레벨업 시스템보다 "동기들이 문제 해결 요청 → 해결하면 보상" 흐름이 스토리와 더 잘 맞았다.

### 이벤트 트리거

이벤트가 언제 발생하냐도 고민이었다.
- 시간 기반
- 처치 수 기반
- 랜덤

시간 기반으로 결정했다. 처치 수 기반은 플레이어가 일부러 안 죽이는 식의 플레이가 생길 수 있고, 랜덤은 너무 불규칙하다.

이벤트 발생하면 게임 일시정지 + 동기 대사. 클리어하면 일시정지 + 보상 선택. 뱀서의 레벨업 UI처럼.

## 최종 확정된 게임 구조

```
무기 선택 → 버그 웨이브 생존
  → 이벤트1 (시간 경과) → 강화 재화 획득
  → 이벤트2 (시간 경과) → 무기 획득
  → 이벤트2 클리어 + 추가 시간 → 장선형 보스전
  → 보스 처치 → 스테이지 클리어
```

버그 종류도 확정했다.
- **SyntaxError**: 직선 이동, 기본형 (빨간 느낌표 외형)
- **NullPointerException**: 빠르고 체력 낮음, 반투명 유령
- **SegFault**: 느리고 체력 높음, 깨진 조각 외형
- **회복 버그**: 도망 다니고, 처치하면 HP 드롭
- **EnvError**: 이벤트 전용, 처치 안 하면 이동속도 감소 장판 생성

## 기술 스택 선택

브라우저에서 돌아가야 한다는 건 확정이었다. 문제는 게임 프레임워크를 쓸지 말지.

- Vanilla JS + Canvas
- Phaser.js
- PixiJS

블로그 시리즈 특성상 외부 라이브러리 없는 게 낫겠다 싶었다. 설명하기도 쉽고, 의존성도 없다. 단, ES6 모듈을 쓰면 `file://`에서 CORS 문제가 생기니까 개발할 땐 `npx serve .` 써야 한다.

스프라이트는 Canvas API로 직접 32×32 픽셀 아트를 그린다.

## MVP 범위

처음엔 여러 스테이지를 한번에 만들고 싶었는데, 냉정하게 생각해보면 1스테이지도 시스템 구현할 게 한둘이 아니다.

**MVP = Stage 1 (파이썬 기초)**

- 이벤트 E1: 들여쓰기 지옥 (IndentationError 15마리)
- 이벤트 E2: 파이참 위기 (EnvError + 60초 생존)
- 퀘스트 Q1: 버그 100마리 처치 → 스탯 업그레이드
- 보스: 장선형

완성하면 나중에 스테이지 추가는 파일 하나 더 만드는 식으로 확장할 수 있게 구조를 짰다.
